# 智能客服自动应答系统 — 技术方案

> 版本: v1.0 | 日期: 2026-04-09 | 基于现有 AI_DEMO 演示项目升级

---

## 一、系统目标

将现有的"人工粘贴消息 → AI辅助生成回复 → 手动复制发送"模式，升级为：

```
客户消息 ──→ [智能分流] ──┬──→ 简单问题 → AI 自动应答 → 自动发送
                          │
                          └──→ 复杂问题 → 进入人工队列 + AI 提供应答策略
```

**核心原则**：简单问题自动化提效，复杂问题AI辅助不误判。

## 三、各模块详细设计

### 3.1 Step 0：智能分流层

**输入**：客户消息原文 + 会话历史（如有）
**输出**：

```json
{
  "complexity": 7,
  "sentiment": "negative",
  "risk_level": "high",
  "needs_order_info": true,
  "decision": "HUMAN",
  "reason": "客户情绪激动且涉及退款金额争议，需人工介入",
  "suggested_strategy": "先安抚情绪，确认订单详情后提出部分退款方案",
  "recommended_script_ids": ["partial-refund-001"]
}
```

**分流规则（LLM + 规则引擎混合）**：

| 条件 | 分流 | 说明 |
|---|---|---|
| 情绪=愤怒 或 风险=高 | → HUMAN | 避免AI误判激化矛盾 |
| 涉及退款金额 > 阈值 | → HUMAN | 需人工决策 |
| 多轮未解决（>3轮） | → HUMAN | AI无法处理 |
| 话术匹配置信度 < 0.7 | → HUMAN | 没有合适的标准回复 |
| 简单查询（物流状态、操作指引） | → AUTO | 标准化程度高 |
| 首次沟通 + 正常情绪 + 有匹配话术 | → AUTO | 最适合自动化 |

**技术实现**：
- Prompt 设计要求 LLM 输出结构化分流 JSON
- 硬规则兜底：金额阈值、轮数限制等用代码实现，不完全依赖 LLM
- 置信度不足时默认走 HUMAN（宁可多人工，不可误自动）

### 3.2 消息接入层

#### 亚马逊 SP-API

```
定时轮询 (每 2-5 分钟)
    │
    ▼
GET /messaging/v1/orders/{orderId}/messages
    │
    ▼
检测新消息 → 写入 DB → 推入处理队列
```

**关键技术点**：
- SP-API 需要 LWA (Login with Amazon) OAuth 认证
- 消息有 API 限流（burst: 5, rate: 0.2/s），需队列控制
- 回复通过 `POST /messaging/v1/orders/{orderId}/messages/confirmDeliveryDetails` 等端点
- 不同消息类型（订单咨询、售后、差评回复）走不同 API 端点

#### 邮件（备选）

```
IMAP 监听 / 轮询
    │
    ▼
解析邮件正文 → 提取客户消息 → 推入处理队列
回复通过 SMTP 发送
```

### 3.3 知识库升级：向量检索（RAG）

**现状问题**：当前把全部话术标题塞进 prompt（catalog stuffing），话术超过 ~50 条后会撞 token 上限。

**升级方案**：

```
话术入库时:
  话术文本 ──→ Embedding 模型 ──→ 向量 ──→ 存入 Qdrant

查询时:
  客户消息 ──→ Embedding ──→ 在 Qdrant 中检索 Top-K ──→ 只把最相关的 5-10 条送入 LLM
```

| 对比 | 现有方案 | RAG 方案 |
|---|---|---|
| 话术规模 | ~50 条上限 | 无上限 |
| 匹配精度 | 依赖 LLM 阅读全文 | 语义相似度预筛 + LLM 精选 |
| Token 消耗 | 高（全量 catalog） | 低（只送 Top-K） |
| 成本 | 高 | 低 |

**Embedding 模型选型**：
- 阿里 DashScope `text-embedding-v3`（和 Qwen 同平台，无需额外 API）
- 或开源 `bge-m3`（多语言支持好，可本地部署）

### 3.5 Step 3：质量校验层（自动回复前）

自动回复发送前的最后一道防线：

```json
{
  "checks": {
    "language_match": true,       // 回复语言与客户一致
    "no_sensitive_words": true,   // 无敏感词/承诺
    "no_hallucination": true,     // 未编造订单号/金额
    "tone_appropriate": true,     // 语气得体
    "variables_filled": true      // 无残留 {{变量}}
  },
  "pass": true
}
```

**不通过 → 自动降级为 HUMAN**

## 四、技术选型

### 4.1 后端

| 组件 | 选型 | 理由 |
|---|---|---|
| **运行时** | Node.js 20 + TypeScript | 与前端技术栈统一，团队学习成本最低 |
| **框架** | NestJS | 模块化、依赖注入、企业级结构 |
| **任务队列** | BullMQ + Redis | 消息处理、限流、重试、延时任务 |
| **数据库** | PostgreSQL | 关系型，事务支持好，适合工单/会话数据 |
| **向量数据库** | Qdrant | 开源、轻量、支持多语言 embedding |
| **缓存** | Redis | 队列复用 + API 缓存 + 限流计数 |
| **ORM** | Prisma | TypeScript 原生，类型安全 |

### 4.2 前端

| 组件 | 选型 | 理由 |
|---|---|---|
| **框架** | React 19 + Next.js 15 | SSR + API Routes，前后端一体化部署可选 |
| **UI** | TailwindCSS + shadcn/ui | 复用现有设计风格 |
| **状态管理** | Zustand | 延续现有方案 |
| **实时通信** | WebSocket (Socket.IO) | 新消息实时推送、队列状态更新 |

### 4.3 LLM 调用层

| 能力 | 实现 |
|---|---|
| **主模型** | Qwen3.5-flash (DashScope) — 快速+低成本 |
| **Fallback 模型** | DeepSeek-chat — 主模型超时/故障时切换 |
| **Embedding** | DashScope text-embedding-v3 |
| **限流** | 令牌桶算法，与平台 API 限流对齐 |
| **重试** | 指数退避，最多 3 次 |
| **超时** | 单次请求 30s 超时 → 降级为 HUMAN |

### 4.4 部署

| 环境 | 方案 |
|---|---|
| **开发/测试** | Docker Compose 一键启动全部服务 |
| **生产** | 阿里云 ECS + RDS(PostgreSQL) + Redis + Qdrant |
| **CI/CD** | GitHub Actions → 自动测试 → Docker 镜像 → 部署 |

## 五、数据模型（核心表）

```sql
-- 会话
CREATE TABLE conversations (
  id            UUID PRIMARY KEY,
  platform      VARCHAR(20),      -- 'amazon', 'email'
  platform_id   VARCHAR(200),     -- 平台侧会话/订单ID
  customer_lang VARCHAR(10),      -- 检测到的语言
  status        VARCHAR(20),      -- 'pending', 'auto_replied', 'human_queue', 'resolved'
  risk_level    VARCHAR(10),      -- 'low', 'medium', 'high'
  assigned_to   UUID REFERENCES users(id),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 消息
CREATE TABLE messages (
  id              UUID PRIMARY KEY,
  conversation_id UUID REFERENCES conversations(id),
  direction       VARCHAR(10),    -- 'inbound', 'outbound'
  content         TEXT,
  chinese_translation TEXT,
  sender_type     VARCHAR(10),    -- 'customer', 'ai', 'human'
  metadata        JSONB,          -- 分流结果、匹配结果等
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 知识库话术
CREATE TABLE scripts (
  id          UUID PRIMARY KEY,
  category    VARCHAR(100),
  title       VARCHAR(200),
  content     TEXT,
  keywords    TEXT[],
  embedding   VECTOR(1024),     -- 或存 Qdrant 中
  is_active   BOOLEAN DEFAULT TRUE,
  version     INT DEFAULT 1,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- AI 处理日志
CREATE TABLE ai_logs (
  id              UUID PRIMARY KEY,
  conversation_id UUID REFERENCES conversations(id),
  step            VARCHAR(20),    -- 'triage', 'match', 'generate', 'quality_check'
  input           JSONB,
  output          JSONB,
  model           VARCHAR(50),
  tokens_used     INT,
  latency_ms      INT,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 操作审计
CREATE TABLE audit_logs (
  id          UUID PRIMARY KEY,
  user_id     UUID REFERENCES users(id),
  action      VARCHAR(50),      -- 'approve_reply', 'edit_reply', 'escalate'
  target_id   UUID,
  details     JSONB,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

## 六、安全设计

| 风险 | 措施 |
|---|---|
| **API Key 泄露** | 所有 LLM/平台 API Key 存服务端环境变量，永不传前端 |
| **越权操作** | JWT 认证 + RBAC 角色权限（管理员、客服、只读） |
| **数据安全** | PostgreSQL 加密存储、传输 TLS、定期备份 |
| **AI 误发** | Step 3 质量校验 + 高风险自动降级 + 人工审核 |
| **注入攻击** | 客户消息在送入 prompt 前做 sanitize |

## 七、关键指标（KPI）

| 指标 | 定义 | 目标 |
|---|---|---|
| **自动应答率** | AI自动回复的消息占比 | 初期 40%，优化后 60%+ |
| **平均响应时间** | 消息到达 → 回复发出 | 自动: <30s，人工: <10min |
| **AI 采纳率** | 人工队列中直接采纳AI草稿的比例 | >70% |
| **误判率** | AI 自动回复但不恰当的比例 | <2% |
| **客户满意度** | 回复后客户反馈 | 不低于纯人工水平 |

## 十、风险与应对

| 风险 | 概率 | 应对 |
|---|---|---|
| 亚马逊 SP-API 审批周期长 | 高 | 先用邮件通道验证全链路，并行申请 SP-API |
| AI 自动回复引发客诉 | 中 | 初期高阈值（仅最简单的走自动），逐步放宽 |
| LLM 响应延迟/不稳定 | 中 | Fallback 模型 + 超时降级为人工 |
| 知识库维护跟不上 | 中 | 数据看板反馈未匹配消息，驱动话术补充 |
