# AI 客服助手本地优先方案

> 版本: v1.0
> 日期: 2026-04-16
> 适用前提: 不引入后端服务、不引入数据库、数据继续使用 `json` 和 `md` 文件存储

---

## 1. 目标

在不显著增加系统复杂度的前提下，解决当前项目的几个核心问题：

- 重新构建后数据容易丢失
- 配置、Prompt、话术库缺少稳定持久化
- 缺少可迁移、可备份、可恢复的数据组织方式
- 想保留未来多场景切换能力
- 不部署云服务，但希望兼容云盘同步

本方案的核心原则：

- 本地优先
- 文件即数据
- 工作区即产品边界
- 云盘兼容，但不内建云同步服务

---

## 2. 方案结论

本项目不做后端、不做数据库时，最佳演进方向不是继续把数据零散存到 Electron 默认目录中，而是正式引入“工作区目录”。

每个工作区都是一个可复制、可备份、可同步的文件夹，里面包含该场景下的全部核心数据：

- 话术库
- Prompt 配置
- 模型配置
- 统计记录
- memory 文件
- 自动备份

应用本身只负责：

- 打开工作区
- 读写工作区文件
- 校验版本
- 生成备份
- 检测外部修改

不负责：

- 登录注册
- 云端鉴权
- 服务端同步
- 多人协作一致性

---

## 3. 产品形态定义

建议把当前产品重新定义为：

**可移植的本地 AI 客服工作台**

它不是 SaaS，不是云平台，而是一个：

- 单用户
- 多工作区
- 本地持久化
- 可导入导出
- 可借助云盘弱同步

的桌面工具。

---

## 4. 核心设计

### 4.1 工作区

工作区是本方案的最小数据单位。

一个工作区对应一个独立场景，例如：

- Amazon-DE
- 退款场景
- 物流咨询
- 配件销售

每个工作区互相隔离，拥有自己的：

- 话术库
- Prompt
- 模型配置
- 统计
- memory
- 备份

### 4.2 工作区目录结构

推荐目录结构如下：

```text
workspace/
  workspace.json
  prompts.json
  providers.json
  ui-settings.json
  knowledge-base.json
  stats/
    sessions.jsonl
  memory/
    CONSTITUTION.md
    SOUL.md
    SKILLS.md
    SCRIPTS_FEEDBACK.md
    HEARTBEAT.md
    daily/
      2026-04-10.md
  backups/
  exports/
```

### 4.3 文件职责

`workspace.json`

- 工作区名称
- 工作区版本
- 创建时间
- 最后更新时间
- 默认语言
- 当前激活场景说明

`prompts.json`

- Prompt A
- Prompt B
- 其他可配置模板

`providers.json`

- LLM 提供商列表
- 当前激活模型
- 相关参数

`ui-settings.json`

- 窗口偏好
- 紧凑模式
- 透明度
- 最近面板状态

`knowledge-base.json`

- 话术库主体
- 分类
- 关键词
- 删除状态
- 版本号

`stats/sessions.jsonl`

- 一行一条会话记录
- 比单个大 JSON 更抗损坏
- 更适合持续追加

`memory/*.md`

- Agent 记忆与每日日志

`backups/`

- 自动备份快照

## 5. 数据格式建议

### 5.1 统一规则

所有 JSON 文件建议统一遵守以下规则：

- UTF-8 编码
- 顶层包含 `version`
- 顶层包含 `updatedAt`
- 写入时尽量原子化

示例：

```json
{
  "version": "1.0.0",
  "updatedAt": "2026-04-16T10:30:00.000Z",
  "data": {}
}
```

### 5.2 统计格式

统计建议改为 `jsonl`：

```jsonl
{"timestamp":"2026-04-16T09:10:00.000Z","intent":"refund","language":"de","decision":"HUMAN","matched":true,"edited":true}
{"timestamp":"2026-04-16T09:15:00.000Z","intent":"shipping","language":"fr","decision":"AUTO","matched":true,"edited":false}
```

原因：

- 追加写简单
- 单条损坏不会拖垮整个文件
- 后续做统计扫描更稳定

## 6. 存储位置建议

不建议继续把全部核心数据仅放在 Electron 默认 `userData` 目录中。

建议改为：

- 用户首次启动时选择一个“工作区根目录”
- 默认推荐位置为用户文档目录

推荐示例：

```text
Documents/AI-CS-Workspaces/
  Amazon-DE/
  Refund-Scenario/
  Logistics/
```

应用内只保存：

- 最近打开的工作区路径
- 当前工作区路径

这样做的好处：

- 用户可见
- 易于复制
- 易于备份
- 易于放入 OneDrive/Dropbox/坚果云

---

## 7. 同步策略

本方案不自建云同步，但允许通过外部工具实现弱同步。

### 7.1 手动同步

最简单也最稳的方式：

- 导出工作区为 zip
- 在另一台机器导入

适用场景：

- 偶尔迁移
- 备份恢复
- 不追求实时同步

### 7.2 云盘目录同步

用户把工作区目录放在以下位置之一：

- OneDrive
- Dropbox
- iCloud Drive
- 坚果云

应用只负责正常读写文件，云盘负责跨设备同步。

这是本方案下最推荐的“伪云端”模式。

### 7.3 Git 同步

适合技术用户，可选支持：

- 工作区目录初始化为 Git 仓库
- 手动提交与回滚
- 通过 Git 做版本管理

但不建议把 Git 作为普通用户默认方案。

## 8. 必做能力

### 8.1 工作区管理

新增能力：

- 新建工作区
- 打开已有工作区
- 最近工作区列表
- 切换工作区
- 重命名工作区

### 8.2 导入导出

新增能力：

- 导出整个工作区为 zip
- 从 zip 导入工作区
- 从已有目录直接打开

### 8.3 自动备份

建议在以下操作后自动备份：

- 修改 Prompt
- 修改 provider 配置
- 编辑知识库
- 写入 memory

备份规则建议：

- 每日首次写入生成一份日备份
- 关键修改前生成临时快照
- 保留最近 20 份

### 8.4 版本迁移

每个核心文件都带 `version`，应用启动时执行 migration。

至少支持：

- 老版本字段补齐
- 字段重命名
- 缺省值回填

### 8.5 外部修改检测

如果用户通过云盘同步或手工编辑文件，应用需要：

- 检测文件最后修改时间
- 提示“工作区文件已被外部修改”
- 支持重新加载

---

## 9. 不做账号系统时的边界

在本方案下，不应设计真正的账号系统。

最多只保留“本机身份”概念，例如：

- 用户昵称
- 默认签名
- 偏好语言

它不是：

- 注册
- 登录
- 云端身份
- 跨设备账号

如果未来真的要做账号，那已经超出本方案边界。

---

## 10. 安全建议

### 10.1 普通配置

以下数据可以直接文件化存储：

- Prompt
- UI 配置
- 话术库
- 统计
- memory

### 10.2 敏感信息

`API Key` 不建议和普通配置完全一样处理。

最低建议：

- 导出工作区时默认不包含 API Key
- 提供“清除本机密钥”功能

更稳的建议：

- `providers.json` 只存 provider 元信息
- API Key 使用系统安全存储保存

如果短期不做系统安全存储，也至少要明确区分：

- 可导出的工作区数据
- 不应默认导出的本机敏感信息

## 11. 工程改造建议

### 11.1 当前问题

当前项目的数据读写分散在：

- settings
- knowledge base
- memory
- stats
- Electron IPC

这会导致：

- 难以切换数据根目录
- 难以统一备份
- 难以做导入导出

### 11.2 重构目标

引入统一的数据访问层：

- `workspace-manager`
- `settings-repository`
- `knowledge-repository`
- `memory-repository`
- `stats-repository`
- `backup-manager`

核心思路：

- UI 不直接关心文件路径
- Store 不直接拼接磁盘路径
- 所有读写都通过当前工作区上下文完成

### 11.3 推荐模块划分

`electron/main.ts`

- 只负责底层文件访问、压缩导入导出、系统路径能力

`src/lib/workspace/`

- 工作区路径解析
- 文件 schema 校验
- migration
- 外部修改检测

`src/stores/`

- 只负责状态，不直接决定存储路径

## 12. 开发顺序

### Phase 1: 工作区底座

目标：

- 引入工作区目录概念
- 所有核心数据改为从工作区读写

任务：

- 增加 `workspace.json`
- 增加当前工作区路径管理
- 统一数据根目录
- 调整 settings / knowledge / memory / stats 的读写入口

### Phase 2: 导入导出与备份

目标：

- 解决迁移与恢复问题

任务：

- 导出工作区 zip
- 导入工作区 zip
- 自动备份
- 最近备份列表
- 一键恢复

### Phase 3: 多工作区

目标：

- 支持不同客服场景隔离

任务：

- 新建工作区
- 切换工作区
- 最近工作区列表
- 工作区信息页

### Phase 4: 云盘兼容优化

目标：

- 让用户能把工作区放进云盘目录

任务：

- 外部修改检测
- 冲突提示
- 重新加载
- 可选只读保护提示

## 13. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 文件损坏 | 单个 JSON 被异常中断写坏 | 使用临时文件写入后原子替换 |
| 数据膨胀 | 统计和 daily 日志越来越大 | stats 改 jsonl，daily 分文件存储 |
| 云盘冲突 | 多设备同时写同一文件 | 先提示冲突，不做自动合并 |
| 用户误删 | 手动删除工作区文件 | 自动备份 + 导出包 |
| 结构升级失败 | 新版程序读旧文件失败 | 每个文件带 version + migration |

---

## 14. 最终建议

如果坚持“不上后端、不上数据库、不部署服务”，那最合理的长期方案不是继续沿用“零散本地配置”，而是：

**把整个应用正式升级为“工作区驱动的文件化桌面工具”。**

最重要的不是多做功能，而是先把下面四件事做稳：

1. 工作区目录
2. 统一数据访问层
3. 导入导出与自动备份
4. 云盘兼容

这样做之后，你依然保持系统简单，但产品会从“一个 demo”升级为“可长期使用的本地工具”。
