# 智能客服 Agent 系统 — 基于 OpenClaw 思想的架构方案

> 版本: v2.0 | 日期: 2026-04-09
> 在 v1.0 技术方案基础上，引入 OpenClaw 的长时记忆、自主执行、自我进化能力

---

## 一、核心理念

**从"工具"到"Agent"的本质转变**：

| | 现有 Demo (工具) | 目标系统 (Agent) |
|---|---|---|
| **记忆** | 无（每次对话独立） | 分层长时记忆，越用越聪明 |
| **执行** | 被动（人粘贴→AI回复） | 自主（监听→分流→回复→学习） |
| **进化** | 静态（prompt 不变） | 自主进化（心跳反思+经验积累） |
| **协作** | 单体 | 多 Agent 分工协作 |

OpenClaw 核心哲学：**"Text > Brain. Write it down."** — 所有经验写成文件持久化，而非依赖 LLM 的"记忆"。

## 二、Agent 记忆系统

### 2.1 分层记忆架构

```
memory/
├── CONSTITUTION.md          # 🔒 宪法层（不可自动修改）
│   └── 核心业务规则、安全底线、合规要求
│
├── SOUL.md                  # 🔄 三观层（Agent 可自主调整）
│   └── 客服风格、语气偏好、用词习惯
│
├── SKILLS.md                # 🔄 技能层（Agent 可自主更新）
│   └── 各语言回复惯例、平台操作要点、工具使用经验
│
├── SCRIPTS_FEEDBACK.md      # 🔄 话术反馈（自动积累）
│   └── 哪些话术好用、哪些需要改进、匹配偏差记录
│
├── daily/                   # 📝 每日日志（自动生成）
```

### 2.2 各层详细设计

#### CONSTITUTION.md — 宪法（只读）

```markdown
# 客服 Agent 宪法

## 绝对禁止
- 不得自动处理退款金额超过 50 欧元的请求
- 不得向客户承诺不确定的时间节点
- 不得在客户情绪激动时使用自动回复
- 不得修改或删除本文件

## 必须人工的场景
- 客户明确要求"转人工"
- 涉及法律、诉讼、律师相关内容
- 同一客户 3 轮内未解决
- 差评回复（公开可见）

## 合规要求
- 所有回复必须包含正式称谓
- 退货政策必须引用当前版本（v2.3）
- GDPR：不得在回复中暴露其他客户信息
```

Agent 可以读取但**绝对不能修改**此文件。代码层面通过文件权限 + 写入拦截保证。

#### SOUL.md — 三观（可学习）

```markdown
# 客服风格人设

## 基础风格
- 专业但亲切，不过分热情
- 简洁直接，不啰嗦（用户纠正了 3 次）
- 先解决问题，再表达歉意（而非反过来）

## 各语言风格偏好
- 法语：使用 vouvoiement（您），正式商务风格
- 德语：直接坦诚风格，德国客户不喜欢过度道歉
- 西班牙语：稍微热情一些，可以用感叹号
- 英语：中性专业风格

## 上次更新
- 2026-04-08: 法语客户更偏好 "Cher/Chère client(e)" 而非 "Bonjour"
- 2026-04-07: 德语 "Mit freundlichen Grüßen" 比 "Beste Grüße" 更正式
```

Agent 通过日常交互中的人工修改反馈，**自主更新**此文件。

#### SKILLS.md — 技能（可学习）

```markdown
# 客服技能库

## 亚马逊平台操作
- FBA 退货：必须引导客户联系亚马逊，卖家无法直接操作
- FBM 退货：发送退货标签，地址用 RETURN_ADDRESS 配置

## 话术匹配经验
- "batterie ne tiens pas la charge" → 电池故障，非配件请求（2026-04-08 学到）
- 客户说 "Ersatzteil" (德语备件) → 先确认是什么备件再匹配
- "remboursement" 不一定是退款，可能只是询问退款进度

## 回复生成技巧
- 包含订单号的回复：先验证订单号格式（Amazon: xxx-xxxxxxx-xxxxxxx）
- 日期变量：法语用 "le 3 avril 2026"，德语用 "3. April 2026"
```

```markdown
## 统计

- [10:41] FR | 退款争议 | 人工 → AI建议"部分退款" → 人工修改金额后发送
```

### 2.3 上下文动态拼装

每次处理新消息时，Agent 看到的上下文是**运行时拼装**的：

```
┌─ 拼装顺序（三明治结构，重要内容在首尾）──────────────┐
│                                                       │
│  1. CONSTITUTION.md (宪法 — 置顶，最高权重)           │
│  2. SOUL.md (风格人设)                                │
│  3. SKILLS.md (技能经验) — 仅检索相关段落             │
│  4. today.md 今日日志摘要 (近期上下文)                │
│  5. SCRIPTS_FEEDBACK.md 中相关条目                    │
│  6. ───── 当前消息 ─────  (用户输入，放中间偏后)      │
│  7. 匹配到的话术内容 (RAG 检索结果)                   │
│  8. CONSTITUTION.md 关键规则重复 (尾部强化)           │
│                                                       │
└───────────────────────────────────────────────────────┘
```

**关键**：不是全部加载，而是根据当前消息**检索相关片段**。SKILLS.md 可能有几百条经验，只取语义最相关的 5-10 条注入上下文。

### 3.2 学习机制（自动经验积累）

**触发条件**：人工修改了 AI 草稿后发送

```
AI 草稿:
  "Sehr geehrter Kunde, wir entschuldigen uns vielmals für die
   Unannehmlichkeiten. Wir möchten Ihnen eine Teilrückerstattung
   von 15% anbieten..."

人工最终版:
  "Sehr geehrter Kunde, es tut uns leid, dass das Produkt nicht
   Ihren Erwartungen entspricht. Wir bieten Ihnen eine
   Teilrückerstattung von 25% an..."

→ Agent 自动分析差异:
  1. 道歉措辞被简化 → 更新 SOUL.md: "德语客户：简短道歉即可"
  2. 退款比例从 15% 调到 25% → 更新 SKILLS.md: "德语区退款建议 25% 起"
  3. 记录到 SCRIPTS_FEEDBACK.md: "部分退款挽留话术 → 德语场景退款比例偏低"
```

**学习流程代码示意**：

```typescript
async function learnFromHumanEdit(
  aiDraft: string,
  humanFinal: string,
  context: MessageContext
) {
  // 只在差异显著时触发学习
  const similarity = cosineSimilarity(embed(aiDraft), embed(humanFinal))
  if (similarity > 0.95) return  // 差异太小，不值得学习

  const lesson = await llm.call({
    system: `你是一个客服经验分析师。对比AI草稿和人工修改后的最终版本，
             提炼出可复用的经验教训。输出JSON格式：
             {
               "skill_update": "要添加到SKILLS.md的经验（如有）",
               "soul_update": "要更新到SOUL.md的风格偏好（如有）",
               "script_feedback": "对话术匹配的反馈（如有）"
             }`,
    user: `语言: ${context.lang}
           场景: ${context.intent}
           AI草稿: ${aiDraft}
           人工最终版: ${humanFinal}`
  })

  // 追加到对应文件
  if (lesson.skill_update) appendToFile('memory/SKILLS.md', lesson.skill_update)
  if (lesson.soul_update) appendToFile('memory/SOUL.md', lesson.soul_update)
  if (lesson.script_feedback) appendToFile('memory/SCRIPTS_FEEDBACK.md', lesson.script_feedback)

  // 记录到今日日志
  appendToDaily(`💡 从人工修改中学习: ${lesson.skill_update || lesson.soul_update}`)
}
```

## 四、心跳机制（自主进化）

### 4.1 心跳任务

```markdown
# HEARTBEAT.md — 心跳任务清单

## 每 30 分钟
- 检查待处理队列，超过 15 分钟未处理的消息发告警

## 每天 23:00
- 总结今日处理日志（daily/today.md）
- 统计：自动回复率、人工采纳率、匹配偏差
- 发现重复模式 → 更新 SKILLS.md
- 高频失败的话术 → 更新 SCRIPTS_FEEDBACK.md

## 每周日 10:00
- 回顾本周所有 daily/*.md
- 蒸馏：提炼本周学到的风格偏好 → 精简更新 SOUL.md
- 蒸馏：合并重复的技能条目 → 精简 SKILLS.md
- 清理：过期的日志压缩归档
- 生成周报摘要
```

### 4.2 蒸馏机制（周期性精简记忆）

```
本周日志中出现了 7 次类似修改：
  "法语回复中，人工把 Bonjour 改为 Cher client"
  "法语回复中，人工添加了 Cordialement 结尾"
  "法语回复中，人工改用 vous 代替 tu"
  ...

→ 心跳触发蒸馏:
  SOUL.md 新增/更新：
  "法语客户：始终使用 vouvoiement，开头用 Cher/Chère client(e)，
   结尾用 Cordialement 或 Bien cordialement"

→ 同时清理 SKILLS.md 中这 7 条零散记录，合并为 1 条
```

**为什么需要蒸馏**：随着时间推移，SKILLS.md 会膨胀。不蒸馏 → 文件太大 → token 超限 → Agent "健忘"。蒸馏 = 合并去重 + 提升抽象层级。

### 4.3 记忆容量管控

```
CONSTITUTION.md    — 固定 ~500 tokens （不变）
SOUL.md            — 上限 1000 tokens （超限时蒸馏精简）
SKILLS.md          — 上限 3000 tokens （超限时蒸馏合并）
SCRIPTS_FEEDBACK   — 上限 2000 tokens （超限时淘汰旧条目）
daily/today.md     — 上限 2000 tokens （当日活动记录）
daily/yesterday.md — 上限 1000 tokens （仅加载摘要）

总上下文预算 ≈ 10,000 tokens（留给话术内容和生成的空间）
```

### 5.2 Agent 间协商

```
Agent 0 (调度): "这条消息分类为'电池故障'，复杂度 4，建议自动回复"
Agent 1 (应答): "匹配到'电池备件缺货'话术，但客户要求两块电池，话术只提供整机替换"
Agent 1 → Agent 0: "需要确认：客户要求的是电池，我们只能提供整机，这算复杂场景吗？"
Agent 0: "客户需求与方案不完全匹配，升级为 HUMAN，同时生成草稿供参考"
→ 状态: AUTO → HUMAN（带草稿+说明）
```

## 六、技术实现要点

### 6.1 记忆存储

| 记忆类型 | 存储方式 | 理由 |
|---|---|---|
| 宪法/三观/技能 | Git 仓库中的 .md 文件 | 版本控制、diff 可追溯、人工可审查 |
| 每日日志 | Git 仓库 + DB 备份 | 文件方便 Agent 读写，DB 方便统计 |
| 话术 embedding | Qdrant | 语义检索 |

**为什么用 Markdown 文件而非全部入库**：
- Agent 直接读写文件，实现简单
- Git 版本控制 → 可以回滚 Agent 的"错误学习"
- 人工可以直接编辑审查
- 符合 OpenClaw "Text > Brain" 哲学

### 6.2 安全边界

```
                    ┌─────────────────┐
                    │ CONSTITUTION.md │
                    │   🔒 只读       │
                    └────────┬────────┘
                             │ 代码层强制
                    ┌────────▼────────┐
Agent 写入拦截器 ──→│ 文件写入 API     │
                    │                 │
                    │ • 禁止写 CONST* │
                    │ • SOUL/SKILLS   │
                    │   单次追加≤200字│
                    │ • 每日写入次数  │
                    │   上限 50 次    │
                    └─────────────────┘
```

### 6.3 与 v1.0 方案的关系

v2.0 **包含** v1.0 的全部内容，额外增加：

| v1.0 模块 | v2.0 升级 |
|---|---|
| Step 0 分流 | → Agent 0，带记忆上下文 |
| Step 1 匹配 + Step 2 生成 | → Agent 1，注入经验记忆 |
| Step 3 质检 | → Agent 2，独立角色 |
| 静态 prompt | → 动态上下文拼装 |
| 无记忆 | → 分层记忆系统 |
| 无学习 | → 学习者 Agent + 心跳蒸馏 |
| 后端架构不变 | NestJS + PostgreSQL + Redis + Qdrant 不变 |

## 八、风险补充

| 风险 | 应对 |
|---|---|
| **记忆膨胀导致 Agent "健忘"** | token 硬上限 + 定期蒸馏 + 相关性检索而非全量加载 |
| **Agent 学到错误经验** | Git 版本控制，可回滚；CONSTITUTION 锁死底线 |
| **蒸馏丢失重要细节** | 蒸馏前归档原文；关键经验标记 `[PINNED]` 不被蒸馏 |
| **多 Agent 协商死循环** | 最大协商轮数限制（3 轮），超时自动走 HUMAN |
| **LLM 调用成本增加** | 心跳空闲时跳过；分流 Agent 用轻量模型；学习 Agent 批量处理 |
