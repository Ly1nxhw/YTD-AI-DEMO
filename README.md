# AI 客服助手

基于 `Electron + React + TypeScript` 的本地优先桌面应用，用于跨境电商客服回复生成、话术沉淀、工作区管理和知识库迭代。

## 项目定位

这个项目不是一个“内置固定话术和固定模型配置”的单体工具，而是一个以工作区为核心的数据化客服系统：

- 所有业务数据都归属于工作区，而不是归属于安装包。
- 每个工作区都可以独立维护自己的知识库、LLM 配置、Prompt、记忆文件、统计数据和学习记录。
- 打包后的程序默认不携带你的业务话术、素材库和模型配置，适合给不同团队、不同品牌、不同店铺独立使用。
- 系统目标不是一次性生成回复，而是持续积累可复用的话术资产，让知识库随着真实客服对话不断增长。

## 当前支持的功能

### 1. 工作区体系

系统已经实现完整的工作区概念，定位接近 VS Code 的工作区，但更偏向业务数据隔离。

支持能力：

- 首次启动时进入工作区选择流，而不是默认加载内置数据。
- 新建工作区时可以自定义名称和保存路径。
- 支持切换最近工作区。
- 支持打开已有工作区目录。
- 支持重命名当前工作区。
- 支持导出工作区为 ZIP。
- 支持从 ZIP 导入工作区。
- 支持工作区备份和恢复。
- 支持检测工作区外部文件变动，并手动重新加载。

一个工作区当前会包含以下核心内容：

- `workspace.json`：工作区元信息。
- `prompts.json`：Prompt 配置。
- `providers.json`：LLM Provider 配置。
- `ui-settings.json`：界面偏好配置。
- `knowledge-base.json`：知识库。
- `memory/`：记忆系统文件。
- `stats/sessions.jsonl`：使用统计。
- `stats/learning-sessions.jsonl`：从对话学习的话术学习记录。
- `backups/`：工作区备份快照。
- `exports/`：导出产物目录。

### 2. 本地优先的数据存储

项目已经从“安装包内置配置”转为“工作区文件化存储”。

这意味着：

- LLM 配置不再固化在程序安装产物里。
- 话术库不再固化在程序安装产物里。
- Prompt 不再固化在程序安装产物里。
- 记忆文件不再固化在程序安装产物里。
- 打包产物更像一个空壳能力平台，真正的数据由用户导入或创建工作区后生成。

这样做的好处是：

- 交付时不会把你的内部业务配置直接打进 EXE。
- 不同客户或不同项目可以带着自己的工作区独立迁移。
- 工作区可以直接做备份、导出、拷贝和归档。

### 3. LLM 配置管理

系统已经支持按工作区管理模型服务配置。

支持能力：

- 支持 OpenAI-compatible 接口。
- 支持配置多个 Provider。
- 支持切换当前激活 Provider。
- 支持配置 API 地址、API Key、模型名、温度、最大 Token。
- 支持单独覆写 Step 1 / Step 2 使用的模型。
- 支持连接测试。
- 通过 Electron 主进程代理请求，减少前端直接跨域请求带来的问题。

适用的服务类型包括：

- OpenAI-compatible 平台。
- 本地或内网代理的模型服务。
- Ollama 兼容接口。
- 其他遵循 `/chat/completions` 协议的服务。

### 4. 客服回复生成工作流

项目当前已经具备完整的客服回复主链路。

工作流包括：

1. 输入客户消息。
2. 进行 Step 0 / Step 1 分析。
3. 识别语言、意图、关键词和匹配话术。
4. 从知识库中查找可复用条目。
5. 生成客服回复。
6. 输出目标语言回复和中文对照。

当前支持的相关功能：

- 支持多语言客服场景。
- 支持意图分析和话术匹配。
- 支持基于知识库的话术复用，而不是完全自由生成。
- 支持未命中知识库时的兜底生成。
- 支持流式回复。
- 支持剪贴板复制结果。

### 5. 剪贴板与桌面效率能力

这个项目不仅是一个页面工具，也包含桌面辅助能力。

支持能力：

- 剪贴板监听。
- 当剪贴板出现新文本时自动推送到应用。
- 全局快捷键唤起窗口并读取剪贴板内容。
- 窗口置顶。
- 小窗模式。
- 透明度调节。

这些能力适合与电商后台、聊天后台、工单后台并排使用。

### 6. 知识库管理

知识库已经具备基本可用的管理能力，且数据归属于工作区。

支持能力：

- 以结构化 JSON 形式存储知识库。
- 维护分类列表。
- 新增知识条目。
- 批量新增知识条目。
- 修改知识条目。
- 软删除知识条目。
- 按分类筛选。
- 按标题、正文、关键词搜索。
- 导入 Markdown 转换为知识库。

知识条目当前不仅包含基本字段，还包含学习元数据：

- `sourceType`
- `sourceRef`
- `qualityScore`
- `reviewStatus`
- `updatedByLearning`
- `usageCount`
- `lastUsedAt`

这为后续做更细粒度的知识治理打下了基础。

### 7. 工作区初始化

系统已经支持在工作区创建完成后，通过 LLM 为某个业务场景初始化首批知识库。

初始化能力当前支持的输入维度包括：

- 品牌名称
- 产品领域
- 产品概要
- 销售渠道
- 目标市场
- 服务语言
- 履约方式
- 售后策略
- 禁止承诺项
- 客服语气风格
- 重点场景种子

初始化生成结果会输出一批结构化知识条目，适合在没有现成话术库时快速起步。

这部分适合：

- 新品牌从零搭建客服知识库。
- 新类目快速创建第一版话术体系。
- 迁移到新工作区时生成最小可用版本。

### 8. 从对话学习话术

这一块已经从“简单抽取”升级成完整学习工作流，不再只是把 LLM 提取结果直接写入知识库。

当前支持的能力包括：

- 从真实客服对话中抽取候选话术。
- 自动生成标题、分类、关键词、场景和客户表达示例。
- 对候选话术做本地规则校验。
- 给出质量分。
- 标记风险提示。
- 检查是否缺少标题、分类、正文。
- 检查正文是否过短。
- 检查关键词是否不足。
- 检查是否存在过度承诺。
- 检查是否应使用变量占位符。
- 与现有知识库做相似度比对。
- 自动给出“建议新建”或“建议更新已有”。
- 支持人工改写候选内容后重新计算建议。
- 支持逐条选择：
  - 新建
  - 更新已有
  - 忽略
- 支持批量保存。
- 支持把学习行为记录到工作区。

在“更新已有”场景下，当前还支持更新前后差异对比，能够查看：

- 标题变化
- 分类变化
- 场景变化
- 关键词变化
- 正文变化

同时，系统会保存最近学习记录，记录内容包括：

- 学习时间
- 来源摘要
- 候选条目数量
- 新增数量
- 更新数量
- 忽略数量
- 每条候选的处理决策

### 9. 记忆系统

项目保留了分层记忆体系，并已与工作区绑定。

当前记忆目录包括：

- `memory/CONSTITUTION.md`
- `memory/SOUL.md`
- `memory/SKILLS.md`
- `memory/SCRIPTS_FEEDBACK.md`
- `memory/HEARTBEAT.md`
- `memory/daily/*.md`

这套结构适合逐步沉淀：

- 固定原则
- 风格偏好
- 业务经验
- 话术反馈
- 每日记录

### 10. 统计与记录

系统当前已经区分两类记录：

- `stats/sessions.jsonl`
  - 面向主客服处理流程的使用记录。
- `stats/learning-sessions.jsonl`
  - 面向“从对话学习话术”的学习记录。

这样后续做统计分析时，不会把客服回复会话和知识学习行为混在一起。

## 适合的使用方式

目前这个系统比较适合以下场景：

- 跨境电商客服团队的本地桌面辅助工具。
- 需要按品牌、店铺、项目隔离知识资产的团队。
- 希望把 LLM 配置和话术库从安装包中彻底剥离的交付场景。
- 希望通过真实客服对话不断训练和补全知识库的业务团队。

## 目录结构

```text
electron/
  main.ts                Electron 主进程，负责 IPC、工作区文件读写、导入导出、备份恢复
  preload.ts             渲染进程桥接

src/
  components/            主要界面组件
  stores/                Zustand 状态管理
  lib/                   LLM、学习校验、知识合并等基础能力
  agent/                 回复生成与记忆注入逻辑
  types/                 类型定义

docs/
  智能客服系统-技术方案.md
  本地优先_文件化工作区方案.md
```

## 开发命令

```bash
npm install
npm run dev
npm run electron:build
```

说明：

- `npm run dev`：启动前端开发环境。
- `npm run electron:build`：构建前端、Electron 主进程和预加载脚本，并打包 Windows 安装产物。

## 仓库约定

- `docs/` 下保留项目方案文档。
- 根目录业务素材、临时导出文件、打包产物不提交到 Git。
- 如需维护示例素材，建议统一放到单独的 `examples/` 或 `fixtures/` 目录，并明确用途。
